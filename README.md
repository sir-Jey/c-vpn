# VPN Server & Client

Этот проект реализует полноценный VPN-туннель между клиентом и сервером.  
Клиент создаёт виртуальный интерфейс `utun`, маршрутизирует через него весь трафик и отправляет его на сервер через UDP.  
Сервер принимает соединения от нескольких клиентов, создаёт для каждого свой `utun`, пересылает трафик и обеспечивает выход в интернет.

**Проект написан на C, использует libsodium для шифрования (ChaCha20-Poly1305) и работает на macOS (с небольшими изменениями — на Linux).**

## Как это работает

Клиент открывает `utun`, назначает ему IP (`10.0.0.2`) и отправляет весь трафик через зашифрованный UDP-туннель на сервер.  
Сервер получает пакет, проверяет, от какого клиента он пришёл (по IP:порту), создаёт для нового клиента свой `utun` (например, `utun5` с IP `10.0.0.1`), и пересылает трафик.


## Генерация ключа (обязательно!)

И клиент, и сервер должны использовать **один и тот же 32-байтный ключ**.

Сгенерировать его можно отдельной программой или в коде:

```c
#include <sodium.h>
// ...
unsigned char key[crypto_aead_xchacha20poly1305_ietf_KEYBYTES];
crypto_aead_xchacha20poly1305_ietf_keygen(key);
FILE *f = fopen("vpn.key", "wb");
fwrite(key, sizeof(key), 1, f);
fclose(f);
```

## Конфигурация

Скопируй файлы `vpn_serv.conf` и `vpn_cli.conf` в папку с бинарниками.

### Пример `vpn_serv.conf`

```
UDP_PORT = 12346
TUN_IP = 10.0.0.1
MAX_CLIENTS = 256
LOG_FILE = /var/log/vpn.log
```

### Пример `vpn_cli.conf`

```
REMOTE_IP = 127.0.0.1
REMOTE_PORT = 12346
TUN_IP = 10.0.0.2
SERVER_TUN_IP = 10.0.0.1
```

## Принцип работы (обучающая часть)

### 1. Почему UDP, а не TCP?

Потому что TCP внутри TCP вызывает проблемы: потери пакетов вызывают повторные передачи и на внутреннем, и на внешнем уровне. UDP — простой, быстрый и не мешает.

### 2. Что такое TUN?

TUN — виртуальный сетевой интерфейс уровня 3 (IP). Когда программа пишет в него, ядро обрабатывает IP-пакет так, как будто он пришёл из сети. И наоборот — пакеты, идущие через TUN, программа может прочитать.

### 3. Как клиент заставляет весь трафик идти через VPN?

После запуска клиент поднимает TUN и добавляет маршрут по умолчанию через этот интерфейс:

```bash
route add default 10.0.0.2
```

Теперь все пакеты, адресованные не в локальную сеть, уходят в TUN.

### 4. Как работает рукопожатие?

- Клиент отправляет первый UDP-пакет (или просто шифрованный трафик).  
- Сервер получает пакет от незнакомого адреса, создаёт для этого клиента новый TUN и назначает ему IP из внутренней сети.  
- Все последующие пакеты от этого клиента сервер направляет в его личный TUN.

### 5. Почему не используется `listen/accept`?

Потому что UDP — протокол без сохранения состояния. Каждый пакет приходит самостоятельно, и серверу не нужно «принимать соединение». Достаточно запомнить адрес отправителя.

### 6. Как работает маршрутизация на сервере?

Для каждого клиента сервер создаёт отдельный TUN и настраивает его IP (обычно `10.0.0.1` для сервера и `10.0.0.x` для клиента).  
Ядро автоматически создаёт маршрут к клиенту. Для выхода в интернет может потребоваться маскарадинг (NAT), но в этом проекте его нет — только туннелирование.

### 7. Небольшой ликбез: `select` и `poll`

Все сетевые операции не должны блокировать друг друга. Программа использует `poll` (или `select`), чтобы одновременно следить за несколькими файловыми дескрипторами:

- TUN — когда приходит пакет от ядра.
- UDP-сокет — когда приходит пакет от удалённой стороны.

## 📝 Примечания

- Для работы с TUN на Linux нужно изменить `socket(PF_SYSTEM, ...)` на `open("/dev/net/tun", ...)`.  
- В проекте используется ChaCha20-Poly1305 (libsodium) — современное и быстрое шифрование.  
- Это **учебная реализация**, не заточенная на высокую производительность.
